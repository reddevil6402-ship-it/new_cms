The Root Disease

Core Diagnosis
After analyzing 351 Java source files across 28 modules, the core problem is not "large controllers" or "enums in business logic." Those are symptoms.

The root disease is: The system has no abstraction layer between "content concept" and "database table." Every content type is permanently encoded in Java classes, ENUMs, and DDL. Adding a category means a code deployment.

This is the Single Responsibility Violation at the Architecture Level — the architecture itself is responsible for what should be a runtime configuration concern.
02

7 Structural Anti-Patterns Found

P-TTERN 01
ENUM-as-Category Anti-Pattern
25 ENUM files + 22 Convertor files = 47 files that exist only because categories are not database-driven. Adding a category requires recompile → deploy → restart.
P-TTERN 02
Fat Controller Anti-Pattern
Largest controller: 4,571 lines. Average: ~580 lines. Controllers do auth checks, business logic, file I/O, path construction, date parsing, enum conversion — all in one class.
P-TTERN 03
Repository-as-Service Anti-Pattern
Controllers directly inject repositories, bypassing the service layer. 80+ character method names encode entire queries as Java identifiers. The query is invisible to non-Java tooling.
P-TTERN 04
Structural Duplication at Module Scale
All 28 modules contain near-identical 7-file structures. This is not modularity — it is copy-paste architecture. Adding module 29 will look exactly like modules 1–28.
P-TTERN 05
Business Logic in the HTTP Layer
The controller handles date parsing, file upload path construction, parent-child path inheritance, enum conversion, workflow triggers inside a single @PostMapping. No service method exists for "save a record."
P-TTERN 06
Workflow as a Bolted-On Afterthought
27 out of 28 modules independently invoke WorkflowService. No unified audit trail. No single lifecycle owner. Workflow is not a first-class concept — it's an ad-hoc call inside each controller.
P-TTERN 07
Configuration Burned Into Binary
Hardcoded org code "CG57006" in TenderNotice.java. File paths, category IDs, UI layout flags (isGrid, isCard, isNormal) stored as Java class fields. Every variant requires code change or restart.

04

Target Architecture — 3 Axioms

1
Content is data, not code.

- "Tender Notice" with fields -, B, C is not different in kind from a "Vacancy" with fields X, Y, Z. Both are instances of a content type. The content type schema is stored in the database and interpreted at runtime. No Java class exists per content type.

2
Behaviour is pluggable, not hardcoded.

When content transitions between states (draft → review → published), the transition rules, notification targets, and side effects are configured — not compiled. The workflow engine reads configuration; it does not contain content-type-specific if-else chains.

3
Infrastructure is explicit, not implicit.

Every cross-cutting concern (caching, audit, events, search indexing) is handled by an infrastructure layer that content services call through defined ports. There are no surprise side effects inside a save() method.

Microservice Boundaries

Each service boundary is defined by domain cohesion (what changes together) and team autonomy (who owns it), not by technical layer.

Universal content CRUD, lifecycle, versioning. Replaces all 28 module services with a single generic runtime driven by schema-service.
:8086 — SchemaDefinitionDTO

:8082 — WorkflowInstanceDTO

:8083 — MediaAssetDTO

:8087 — FormDefinitionDTO

:8085 — SearchResultDTO

Full-text search, faceted filtering, search index management. PostgreSQL tsvector baseline with OpenSearch for production-scale datasets.
:8088 — NotificationDTO

07

Workflow Engine Design

The workflow engine is a data-driven state machine where states, transitions, guards, and actions are all defined in JSONB configuration stored in cms_workflow.workflow_definitions. No workflow logic exists in content controllers.

Permission Model per Transition

Each transition defines guards (must-pass checks) and actions (side effects after passing). Guards include role checks, field validation, and ownership verification. Actions include notifications, cache invalidation, search indexing, and Kafka event emission.

Adding a New Workflow — Zero Code Change

Need a "Tender-specific 3-level approval" workflow- Open Workflow Designer in CMS Admin → Define states and transitions via UI → Assign to tender_notice content type → Done. Existing content types are entirely unaffected.

Outbox Pattern for Guaranteed Delivery

Every workflow transition writes to an outbox_events table inside the same DB transaction. - Debezium CDC connector or scheduler reads from the outbox and publishes to Kafka. This guarantees exactly-once event delivery even if Kafka is temporarily unavailable.

08

Dynamic Form Engine

Forms are rendered using JSON Schema + UI Schema on the frontend. The backend stores the schema, validates submissions, and routes them to configured action handlers — no per-form React components or Spring controllers needed.

Form Lifecycle

  1. Design — Admin creates form schema in UI (WYSIWYG field builder)
  2. Render — Next.js fetches schema, DynamicForm.tsx renders it automatically
  3. Submit — Server validates against JSON Schema, saves to form_submissions, routes to action handler

Validation

Validation rules live in the schema JSON: required fields, regex patterns (PAN: ^[--Z]{5}[0-9]{4}[--Z]$, GST: full Indian format), min/max, format checks. Server-side validation is identical to client-side — single source of truth.

09

Media Management

The StoragePort interface abstracts all file backends. Switching from local filesystem to AWS S3 requires a configuration change, not a code change. This directly eliminates the hardcoded environment.getProperty("file.repository.path") pattern found across the current system.

Upload Pipeline

  1. Validate (type, size, virus scan)
  2. Generate UUID filename (no original filename in path — prevents path traversal)
  3. Check SHA-256 checksum for deduplication
  4. Store to configured backend (Local / S3 / GCS / Azure Blob)
  5. Generate thumbnail asynchronously via Kafka event
  6. Write metadata to cms_media.media_assets
  7. Return MediaAssetDTO with CDN URL

RBAC & Authorization

The current system uses session-based role checks scattered across 28 controllers. The new system centralises authorization using a Resource-Action-Scope model enforced at the API gateway and service layer, with PostgreSQL Row-Level Security as a final backstop.

Permission Format

content:CREATE:OWN        → Can create content (for themselves)
content:READ:ALL          → Can read all content
content:PUBLISH:DEPT      → Can publish content in own department
tender:APPROVE:ALL        → Can approve any tender
workflow:MANAGE:ALL       → Can configure workflows
media:DELETE:OWN          → Can delete own media

Enforcement Points

HTTP Request → API Gateway (JWT validation) → Service Layer @PreAuthorize → Domain Service (predicate) → Repository (tenant isolation, soft-delete filter)

PostgreSQL Row-Level Security

RLS policies enforce tenant isolation at the database level, making cross-tenant data leaks impossible regardless of application-layer bugs.

Event-Driven Communication

Kafka Topic Registry

cms.content.created       → search-service indexes, audit-service logs
cms.content.updated       → search-service re-indexes, cache invalidation
cms.content.published     → notification-service notifies, CDN purge
cms.content.archived      → search-service removes, analytics event
cms.workflow.transitioned → notification-service, audit-service
cms.media.uploaded        → thumbnail generation, virus scan
cms.user.login            → audit-service, anomaly detection
cms.user.role.changed     → audit-service, cache invalidation
cms.schema.changed        → all services refresh schema cache

Event Schema (CloudEvents 1.0 Standard)

All events follow the CloudEvents specification with a standard envelope: specversion, type, source, id, time, tenantid, correlationid, and typed data payload. This enables replay, dead-letter queuing, and cross-system correlation via trace ID.

user:{id}:permissions                  TTL:  5 min → permission bitset
workflow:{code}:def                    TTL: 30 min → workflow definitions
search:facets:{type}                   TTL: 10 min → aggregation results

Event-Driven Cache Invalidation

When cms.content.published fires, the cache invalidator evicts exactly the affected content item and its listing page entries. This replaces the ad-hoc @Scheduled cache refresh found in the current system with targeted, event-driven invalidation — no more stale data from missed scheduler runs.

Monitoring & Observability

Three Pillars

Metrics — Micrometer → Prometheus → Grafana. Custom metrics: cms_content_create_total, cms_workflow_transition_total, cms_cache_hit_ratio, cms_kafka_consumer_lag.

Traces — OpenTelemetry auto-instrumentation → Jaeger / Tempo. Every HTTP request carries a trace ID across all service calls. "Why did this tender publish fail-" → single trace shows the complete call graph.

Logs — Structured JSON via Logback → Fluentd → Elasticsearch → Kibana. Every log line includes traceId, tenantId, userId, contentId. Never raw string concatenation.

CI/CD Strategy

Zero-downtime deployments are possible because DB migrations run via Flyway before deploy, services are stateless (session in Redis), and Kubernetes rolling updates use health check gates. All migrations must be backward-compatible with the previous service version.

Pipeline stages: Validate (2 min) → Test parallel (10 min) → Docker build (5 min) → Auto-deploy to Dev → Manual promote to Staging → Manual approval gate → Blue/Green to Production → 15-minute observation window with auto-rollback.

15

Migration Strategy (Strangler Fig)

The migration uses the Strangler Fig pattern — the new system grows around the old one, gradually taking over traffic until the old system can be safely decommissioned. No big-bang rewrite.

4 Phases (12–16 weeks)

Phase 1 (Weeks 1–4) — Foundation: Deploy K8s cluster, Kafka, Redis. Deploy IAM, Schema, Audit services. Migrate users and roles. Run both systems in parallel.

Phase 2 (Weeks 5–8) — Content Engine: Map all 25 ENUMs to database category rows. Deploy Content Service with dual-write. Run migration SQL for all 28 module tables into content_items. Verify data parity.

Phase 3 (Weeks 9–12) — Traffic Cutover: Route 10% → 25% → 50% → 100% read traffic to new stack using feature flags. Remove old module controllers one by one.

Phase 4 (Weeks 13–16) — Decommission: Remove all 47 ENUM/Convertor files. Archive old tables. Decommission old application servers. Capture performance baseline.

Architecture Decisions & Rationale

JSONB body for content
Eliminates per-content-type tables. Schema flexibility without EAV performance penalty. PostgreSQL GIN index enables fast queries on any field.
⚠ Harder to write raw SQL. Mitigated by search service and typed query builders.
Hexagonal Architecture
Domain logic has zero framework dependencies. Business rules are testable without Spring, JPA, or HTTP mocking. Domain model is the stable core.
⚠ More boilerplate than traditional layered arch. Justified by testability gains.
Outbox Pattern
Guarantees event delivery even when Kafka is down. Eliminates dual-write consistency bugs. Enables event replay for new consumers.
⚠ Requires a CDC connector or scheduler. Small operational overhead.
Workflow as JSONB config
New approval processes require zero code. Configurable from admin UI. Eliminates repeated workflow code across 27 controllers.
⚠ More complex execution engine. Well-justified by eliminating 27 independent workflow call sites.
PostgreSQL RLS
Tenant data isolation enforced at DB level. Impossible to accidentally leak across tenants regardless of application bugs.
⚠ Requires setting session variables before every query. Handled transparently in the repository layer.
UUID Primary Keys
Safe for distributed generation across services. Opaque to clients (no enumeration attacks). No sequence coordination needed.
⚠ Slightly larger than BIGINT. UUIDv7 (time-ordered) mitigates index fragmentation.
Soft Deletes via deleted_at
Audit trail preservation. Recoverable from accidental deletes. Enables "trash" UX pattern.
⚠ More complex queries (always filter deleted_at IS NULL). Mitigated by PostgreSQL views and RLS policies.
Kafka Async Communication
Services are decoupled. search-service can fail without blocking content-service. Supports event replay and new consumer addition without producer changes.
⚠ Operational complexity. Mitigated by managed MSK or Confluent Cloud.
Summary: The Open/Closed Principle at Architecture Scale
Before: Adding "Grievance Portal" required copying 7 Java files, writing 2 ENUMs + 2 Convertors, a DB migration, if-else additions in 27 workflow controllers, and a deployment with downtime risk.

After: Admin → Content Types → Create New → Add fields → Select workflow → Save → Available immediately. No code. No deployment. No downtime.

The system is open for extension through configuration, closed for modification through code.
